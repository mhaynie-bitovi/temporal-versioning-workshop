## Opening (10 min)

*Goal: Frame the entire workshop as one story with rising stakes. Make the audience feel the problem before showing solutions.*

### P1 - The problem with versioning durable execution (~4 min)
- Temporal workflows can run for days, weeks, years. You will inevitably need to change code while workflows are running.
- If you deploy changed workflow code and a worker replays an in-flight workflow, the commands generated by the new code may not match the events in the history.
- Result: Non-determinism error. Workflow is blocked. Production is broken.
- *"Deploying new code to long-running Workflows has always been one of the trickier problems in Durable Execution."* (Temporal blog)
- Frame it through the valet scenario: "You're the valet parking team. You have a production system. 47 cars are parked right now. Your client wants a new feature. How do you ship it without breaking those 47 in-flight transactions?"

### P1 - The scenario: airport valet parking (~2 min)
- ValetParkingWorkflow: request space -> move car -> sleep (owner's trip) -> move car back -> release space
- ParkingLotWorkflow: long-lived singleton managing 30 spaces, uses continue-as-new
- The sleep is the danger zone: workflows will be "in flight" during deploys
- Keep it brief, just enough to ground the exercises

### P1 - Three versioning strategies (high-level preview) (~3 min)
- Temporal has three strategies for handling non-replay-safe changes to workflow code:
  - **Workflow Type Versioning**: Create a new workflow type (copy the definition, give it a new name). Simplest approach, but doesn't help in-flight workflows.
  - **Workflow Patching**: Use the patching API to branch execution based on markers in the event history. One worker handles both old and new workflows. Works everywhere, but accumulates conditional complexity over time.
  - **Worker Versioning**: Deploy multiple worker versions simultaneously. Temporal routes workflow tasks to the correct version. Eliminates patching for PINNED workflows.
- This workshop focuses on Patching (Exercise 1) and Worker Versioning (Exercises 2-3). The **Worker Controller** (Exercise 3) is not a separate versioning strategy - it's a Kubernetes operator that automates the Worker Versioning deployment lifecycle (rainbow deployments, progressive rollouts, gate checks, sunsetting).
- [DIAGRAM] "The Versioning Spectrum" - single visual showing progression from manual/code-level (type versioning, patching) to infrastructure-managed (worker versioning) to automated (worker controller). This is the mental model for the whole workshop.
- Rising stakes framing: "Over the next 3.5 hours, you're going to ship three releases, each one harder than the last, and handle a production incident."

### P2 - "Where are we" progress slide (reuse throughout)
- Simple visual showing 3 exercises with the current one highlighted
- Reuse after each exercise to show accumulation

## Exercise 1 Concepts (20 min)

*Goal: Teach the replay mechanism well enough that the audience understands WHY patching exists, not just HOW it works. The replay diagram is the most important visual in this section.*

### P1 - How replay works: commands and events (~7 min)
- **Initial execution**: Worker runs workflow code, generates Commands (ScheduleActivityTask, StartTimer, etc.), server stores them as Events in the history
- **Recovery/replay**: Worker re-executes workflow from the beginning, SDK compares generated Commands against stored Events, uses stored Activity results instead of re-executing
- **If Commands don't match Events = Non-determinism Error = Workflow blocked**
- Command-Event mapping table (keep it to the 3-4 most relevant):
  | Workflow Code | Command | Event |
  |---|---|---|
  | Execute activity | ScheduleActivityTask | ActivityTaskScheduled |
  | Sleep/timer | StartTimer | TimerStarted |
  | Child workflow | StartChildWorkflowExecution | ChildWorkflowExecutionStarted |
  | Complete workflow | CompleteWorkflowExecution | WorkflowExecutionCompleted |
- [DIAGRAM] Replay flow: show a workflow executing, generating commands, storing events. Then show replay: same code re-executes, commands compared to events, match = continue, mismatch = NDE. This is THE diagram for Exercise 1. Invest time here.
- Source: LMS "History Replay" module (video embed at https://www.youtube.com/embed/K6tDDs5ijFg for Python version), skill core/determinism.md

### P1 - What is a non-replay-safe change? (~5 min)
- Any change that alters the Command sequence for a given input
- **Non-replay-safe (need versioning):**
  - Adding, removing, or reordering activities
  - Adding or removing timers
  - Switching which activity/child workflow is called
  - Changing execution order of activities/timers relative to each other
  - Starting or cancelling child workflows, signals, nexus operations
- **Safe to deploy without versioning:**
  - Changing activity implementation code (activities aren't replayed)
  - Changing activity/child workflow options (timeouts, retry policies)
  - Changing arguments passed to activities or child workflows
  - Changing timer durations (except to/from zero in Python/Go/Java)
  - Adding new signal/query/update handlers (additive)
  - Adding logging or non-command statements
- The two-question test (from LMS): (1) Does this change affect the sequence of Commands? (2) Will there be open executions when you deploy?
- Source: docs.temporal.io/workflow-definition#deterministic-constraints, LMS courses

### P1 - Replay testing: catching problems before production (~3 min)
- Export a completed workflow's history as JSON
- Write a test that replays that history against your current code
- If the code produces different commands than the history, the test fails with an NDE
- This is your safety net before every deploy
- Keep it brief: the exercise will teach the mechanics. The slide just needs to show the pattern.
- Source: Exercise 1 Parts A-C, skill core/determinism.md

### P1 - workflow.patched(): the fix (~3 min)
- Branching at the code level: `if workflow.patched("my-change"):` new code `else:` old code
- New executions: patched() returns True, records marker in history
- Replay with marker: returns True
- Replay without marker: returns False (history predates this patch)
- Three-phase lifecycle (mention, don't dwell): patch in -> deprecate -> remove
- Source: skill python/versioning.md, docs.temporal.io patching

### P2 - "Pause and think" moment
- "You just deployed new code. 47 cars are currently parked. What happens to the workflows that started on the old code?"

### P3 - TDD angle
- Mention briefly: the replay test is the test you write FIRST, then you make the change, watch it fail, patch it, watch it pass. Natural TDD loop.

## Exercise 2 Concepts (30 min)

*Goal: This is the biggest and most important bridge. Teach worker versioning concepts thoroughly enough to set up the Exercise 2 incident (the dramatic peak). End this section building tension.*

### P1 - Why patching isn't enough (~3 min)
- Recap: patching works but accumulates conditional branches over time
- Every non-replay-safe change adds another `if workflow.patched(...)` block
- Long-lived workflows can accumulate layers of patches
- "Your valet system shipped the notification feature fine. But billing is next, and the patching is already adding complexity."
- This is the bridge motivation: "Here's why what you just learned isn't enough"
- Source: Exercise 2 intro, Temporal blog ("patches accumulate into real code complexity and cognitive overhead")

### P1 - How Worker Versioning works (~8 min)
- **Core idea**: Instead of embedding version logic in workflow code, declare versioning behavior at the workflow type level. Temporal's infrastructure handles routing.
- **Worker Deployment**: Logical service grouping (e.g., "valet"). All versions of your code live under this umbrella.
- **Worker Deployment Version**: Specific code snapshot identified by deployment name + Build ID (e.g., "valet:1.0", "valet:2.0")
- **Build ID**: Identifies the code version. Typically a git commit hash, version number, or timestamp.
- **Configuration**: `WorkerDeploymentConfig` with `deployment_name`, `build_id`, `use_worker_versioning=True`
- [DIAGRAM] Worker Deployment containing multiple Worker Deployment Versions, each with workers polling the same task queue but different versions. Temporal Server routes based on routing config.
- Source: docs.temporal.io/worker-versioning, Temporal blog announcement, skill core/versioning.md

### P1 - PINNED vs AUTO_UPGRADE (~5 min)
- Two-column comparison slide (not two separate explanations):
  | | PINNED | AUTO_UPGRADE |
  |---|---|---|
  | Behavior | Stays on original version until completion | Moves to new version on next workflow task |
  | Patching needed? | Never | Yes (replays old history against new code) |
  | Best for | Short-to-medium workflows (minutes to hours) | Long-running workflows (weeks to years) |
  | In our workshop | ValetParkingWorkflow (each parking transaction) | ParkingLotWorkflow (immortal singleton) |
  | Key benefit | Eliminates version compatibility complexity | Keeps long-running workflows on latest code |
- Decision guide (from official docs):
  | Workflow Duration | Uses CaN? | Recommended | Patching? |
  |---|---|---|---|
  | Short (completes before next deploy) | N/A | PINNED | Never |
  | Medium (spans multiple deploys) | No | AUTO_UPGRADE | Yes |
  | Long (weeks to years) | Yes | PINNED + upgrade on CaN | Never |
  | Long (weeks to years) | No | AUTO_UPGRADE + patching | Yes |
- Source: docs.temporal.io/worker-versioning#choosing-behavior, Temporal blog GA announcement

### P1 - Version lifecycle and routing (~5 min)
- **Version statuses** (one visual showing the progression):
  1. **Inactive** - Version exists (worker polled), but not receiving traffic. Can receive workflows via VersioningOverride only.
  2. **Active** (Current or Ramping) - Accepting new workflows and existing auto-upgrade workflows.
  3. **Draining** - No new workflows, but open pinned workflows still running. Cannot safely decommission.
  4. **Drained** - All pinned workflows completed. Safe to shut down workers (may still need for queries on closed workflows within retention period).
- **Current Version**: Where all new workflow executions go (unless ramping)
- **Ramping Version**: Receives a configurable percentage of new executions. Used for progressive rollouts.
- **Target Version**: For a given workflow, the version it will move to next (relevant for AUTO_UPGRADE)
- `set-current-version` CLI command: tells Temporal which version should receive new traffic
- Source: docs.temporal.io/worker-versioning#versioning-statuses, worker-controller concepts.md

### P1 - Rainbow deployments: why multiple versions coexist (~4 min)
- **Rolling**: Workers upgraded in place. No control over routing. Slow rollback. Incompatible with Worker Versioning.
- **Blue-green**: Two versions. Can route traffic between them. Instant rollback. But only two "colors" - not enough for pinned workflows that need their original version kept alive.
- **Rainbow**: Like blue-green but with more colors. New versions deployed alongside old ones. Old versions drain naturally as pinned workflows complete. This is what Worker Versioning enables.
- [DIAGRAM] Rainbow deployment: v1.0 workers handling draining pinned workflows, v2.0 workers handling new workflows, potentially v3.0 being tested. All coexisting. Temporal Server routing traffic.
- "You need rainbow because when you ship a bug, or deploy twice in quick succession, it's easy to need three or more builds around." (Temporal blog)
- Source: docs.temporal.io/worker-versioning#deployment-systems, Temporal blog ("Rolling the dice / Blue-green / Taste the rainbow")

### P1 - Emergency remediation (~3 min)
- **Instant rollback**: `set-current-version` back to previous version. New traffic redirected in seconds. No redeploy needed.
- **Evacuate stuck workflows**: `update-options` with a query to bulk-move workflows from broken version to working version.
- **Fix-forward**: Deploy a fixed version (v3.1), set as current. Incident resolved.
- This is the setup for the Exercise 2 peak moment: "Now let's see what happens when things go wrong."
- Source: Exercise 2 Part C, docs.temporal.io update-options

### P2 - Patching + Auto Upgrade: the catch (~2 min)
- AUTO_UPGRADE workflows still replay old history against new code when they auto-upgrade
- Non-replay-safe changes to AUTO_UPGRADE workflows still require patching
- Exercise 2 Part D demonstrates this: add a timer to ParkingLotWorkflow, get NDE, fix with patch
- "PINNED eliminates patching. AUTO_UPGRADE does not."
- Source: Exercise 2 Part D, docs.temporal.io/worker-versioning#choosing-behavior

### P2 - Upgrade on Continue-as-New ("trampolining") (~2 min)
- For long-running workflows using continue-as-new: upgrade to new version at CaN boundary
- Each run stays pinned (no patching), but the next run starts on the target version
- Workflow detects `target_worker_deployment_version_changed`, opts into upgrade at next CaN
- Ideal for: entity workflows, AI agents, batch processors
- Currently in Public Preview
- Mention conceptually, don't dive deep. One slide max.
- Source: docs.temporal.io/worker-versioning#upgrade-on-continue-as-new, Temporal blog GA announcement, Go SDK v1.41.0 release notes

### P1 - "Pause and think" / tension builder
- "Everything is humming. Billing shipped clean. v1.0 drained on its own. You're feeling good about versioning. Then you make a typo in the tip calculation..."
- Set up the Exercise 2 incident

## Exercise 3 Concepts (15 min)

*Goal: Post-break, low energy. Keep it tight and punchy. Start with a bang: "You're going to automate everything you just did by hand."*

### P1 - "Welcome back. You're about to fire yourself." (~1 min)
- Reframe: "For the last two exercises, you were the ops team. Now you're going to automate everything you just did by hand."
- [SLIDE] Side-by-side: "Exercise 2: 6+ CLI commands, manual monitoring, manual draining" vs "Exercise 3: update a YAML file"

### P2 - Kubernetes in 30 seconds (~1 min)
- Do NOT try to teach Kubernetes. Two sentences:
  - A **Pod** is a running container (your worker process)
  - A **Deployment** manages a set of Pods (scaling, rolling updates)
- If they don't know K8s, 15 minutes won't fix that. Just enough vocabulary to follow along.

### P1 - What the Worker Controller does (~4 min)
- Open-source Kubernetes operator (github.com/temporalio/temporal-worker-controller)
- You declare desired state in a `TemporalWorkerDeployment` CRD. The controller handles:
  - Creating versioned Kubernetes Deployments (one per build ID)
  - Registering versions with Temporal
  - Progressive traffic ramping
  - Running gate workflows before routing production traffic
  - Sunsetting drained versions (scale down + delete)
- **Key relationship**: One `TemporalWorkerDeployment` CRD -> Multiple Kubernetes Deployments (one per version, managed by controller)
- [DIAGRAM] Architecture from worker-controller docs: TWD resource -> controller creates versioned Deployments -> Pods poll Temporal Server with their version
- Source: worker-controller concepts.md, architecture.md, docs.temporal.io/kubernetes-controller

### P1 - Rollout strategies (~4 min)
- **AllAtOnce**: Every replica cuts over immediately. Fine for initial deploy or when no in-flight workflows exist.
- **Progressive**: Ramp traffic gradually. Steps define ramp percentages and pause durations. Old pinned workflows keep running on old version. This is the rainbow deployment model automated.
  ```yaml
  rollout:
    strategy: Progressive
    steps:
      - rampPercentage: 25
        pauseDuration: 30s
      - rampPercentage: 75
        pauseDuration: 30s
  ```
- **Manual**: Controller creates the versioned Deployment but does NOT promote. Version stays Inactive. You test, then promote yourself. Used for pre-deployment testing with synthetic traffic.
- One comparison table, not three separate explanations.

### P1 - Gate workflows (~3 min)
- When `spec.rollout.gate` is configured: controller starts gate workflow on new version while it's still Inactive
- Gate must complete successfully before any production traffic is routed
- If gate fails, version stays Inactive, production is unaffected
- Compare to Exercise 2: "There, a bad deploy hit live traffic before you could respond. Here, the gate catches it before any customer is affected."
- Source: Exercise 3 Part C, worker-controller architecture.md

### P2 - Pre-deployment testing with synthetic traffic (~2 min)
- Manual strategy + VersioningOverride lets you pin test workflows to an Inactive version
- Test runs full code path on real infrastructure (same namespace, same Temporal server, same task queue)
- No special test logic or sandbox needed
- Source: Exercise 3 Part D

## Wrap-up (15 min)

*Goal: Callback to opening, decision framework they can use Monday morning, moment of reflection.*

### P1 - The callback (~2 min)
- "At 1:30, we said deploying changes to durable executions is the fundamental challenge. You just solved it with two versioning strategies, then automated the deployment lifecycle."
- [SLIDE] "Where are we" progress visual with all 3 exercises completed/highlighted

### P1 - Decision framework: which strategy when? (~5 min)
- [DIAGRAM/FLOWCHART] Practical, opinionated decision tree:
  - Do you have running workflows when you deploy? No -> just deploy. Yes -> continue.
  - Is the change non-replay-safe? No -> just deploy. Yes -> continue.
  - Can you run multiple worker versions simultaneously (rainbow)? No -> use patching. Yes -> continue.
  - Are workflows short-lived (complete before next deploy)? Yes -> PINNED, no patching needed. No -> continue.
  - Do workflows use continue-as-new? Yes -> PINNED + upgrade on CaN. No -> AUTO_UPGRADE + patching.
  - Are you on Kubernetes? Yes -> Worker Controller. No -> manual CLI management.
- This is the slide people will photograph. Make it clear and opinionated.

### P1 - Recap: what you learned (~3 min)
- **Replay testing**: Your safety net. Export history, replay against new code, catch NDEs before production.
- **Patching**: Code-level branching. Works everywhere, but accumulates complexity.
- **Worker Versioning**: Infrastructure-level routing. PINNED eliminates patching for short workflows. AUTO_UPGRADE keeps long workflows current but still needs patching.
- **Worker Controller**: Automates rainbow deployments on K8s. Progressive rollouts, gates, sunsetting.
- **Emergency remediation**: Instant rollback (set-current-version), evacuate stuck workflows (update-options), fix-forward.
- Use repetition: recap each concept in one sentence. "Repetition is a feature, not a bug." (speaking.io)

- 30 seconds of silence. Let people think.
